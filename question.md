# Блок 1: Основы языка (Kotlin)

## 1. Классы и объекты

### **Базовые вопросы**

#### **Что такое класс и объект?**
* **Ответ:** 
  * **Класс** — это определяемый пользователем тип данных, логический шаблон и чертеж, описывающий структуру (свойства) и поведение (методы), общие для всех создаваемых экземпляров. На уровне рантайма класс представлен метаданными в сегменте Metaspace (в JVM).
  * **Объект (экземпляр)** — конкретная область памяти, выделенная в куче (Heap) во время выполнения программы (`runtime`), хранящая реальные значения свойств и ссылку на метаданные своего класса.
* **Пример:**
```kotlin
// Класс — абстрактное описание структуры сессии пользователя
class UserSession(val token: String, val userId: String) {
    fun isValid(): Boolean = token.isNotEmpty()
}

// Создание объектов в куче. Каждая переменная указывает на свой уникальный адрес в памяти
val sessionAlice = UserSession(token = "aSdF123", userId = "user_01")
val sessionBob = UserSession(token = "qWeR456", userId = "user_02")
```
* **Важные детали:** В JVM 64-бит каждый обычный объект имеет заголовок (`Object Header`), состоящий из `Mark Word` (8 байт: хэш-код, возраст сборщика мусора, информация о блокировках синхронизации) и `Klass Word` (4 байта при включенном `-XX:+UseCompressedClassPointers`, указывающий на метаданные класса в Metaspace).

---

#### **Чем отличается `init` блок от вторичного конструктора?**
* **Ответ:** 
  * **Блок `init`** является частью первичного конструктора. Код внутри `init` блоков компилируется непосредственно в тело основного конструктора класса и выполняется при его вызове.
  * **Вторичный конструктор (`secondary constructor`)** — это альтернативный способ создания экземпляра. Он обязан делегировать вызов первичному конструктору (напрямую через `this(...)` или косвенно через другой вторичный конструктор). Тело вторичного конструктора выполняется только после того, как отработают все блоки `init` и инициализаторы свойств.
* **Пример:** Настройка кастомного View в Android с обработкой стилей.
```kotlin
class CustomActionButton @JvmOverloads constructor(
    context: Context,
    attrs: AttributeSet? = null,
    defStyleAttr: Int = 0
) : FrameLayout(context, attrs, defStyleAttr) {

    // Шаг 1: Инициализация полей класса и выполнение блока init
    init {
        inflate(context, R.layout.view_custom_action_button, this)
        println("Первичная инициализация: разметка раздута.")
    }

    // Шаг 2: Вторичный конструктор (используется, если объект создается только по контексту)
    constructor(context: Context) : this(context, null) {
        println("Выполнен дополнительный код вторичного конструктора.")
    }
}
```

### **Глубокие вопросы**

#### **Как компилятор Kotlin транслирует первичный конструктор и блоки `init` в Java-байткод?**
* **Ответ:** JVM не знает концепции «блоков `init`». На уровне байткода компилятор Kotlin собирает все объявления свойств, инициализаторы полей и блоки `init` в один или несколько Java-конструкторов (метод `<init>`). Порядок копирования кода строго соответствует порядку их объявления в файле сверху вниз.
* **Пример:**
```kotlin
// Исходный код на Kotlin
class PaymentValidator(val maxLimit: Double) {
    val isEnabled = true
    
    init {
        require(maxLimit >= 0.0) { "Лимит не может быть отрицательным" }
    }
}

// Декомпилированный эквивалент на Java
public final class PaymentValidator {
   private final double maxLimit;
   private final boolean isEnabled;

   public PaymentValidator(double maxLimit) {
      this.maxLimit = maxLimit;
      this.isEnabled = true; // Свойство инициализируется в самом начале конструктора
      
      // Сюда компилятор скопировал тело блока init:
      if (maxLimit < 0.0) {
         throw new IllegalArgumentException("Лимит не может быть отрицательным");
      }
   }

   public final double getMaxLimit() { return this.maxLimit; }
   public final boolean isEnabled() { return this.isEnabled; }
}
```

#### **Каков жизненный цикл инициализации объекта при наследовании?**
* **Ответ:** При создании объекта наследника JVM гарантирует инициализацию всей цепочки наследования снизу вверх на уровне вызовов, но выполнение кода инициализации (полей и блоков) происходит сверху вниз:
  1. Выделяется память под весь объект (включая свойства родителя и наследника). Все поля заполняются дефолтными значениями типов (`null`, `0`, `false`).
  2. Вызывается конструктор дочернего класса, который первым делом делегирует вызов суперконструктору `super()`.
  3. Инициализируются свойства родительского класса и выполняются его блоки `init`.
  4. Выполняется тело конструктора родительского класса.
  5. Инициализируются свойства дочернего класса и выполняются его блоки `init`.
  6. Выполняется тело конструктора дочернего класса.
* **Пример:**
```kotlin
open class BaseLogger(val tag: String) {
    init { println("Родитель: init блок. Тег: $tag") }
}

class FileLogger(tag: String, val filePath: String) : BaseLogger(tag) {
    init { println("Наследник: init блок. Путь к файлу: $filePath") }
}

// Вызов: FileLogger("AppTag", "/sdcard/log.txt")
// Выведет:
// Родитель: init блок. Тег: AppTag
// Наследник: init блок. Путь к файлу: /sdcard/log.txt
```

### **Каверзные вопросы**

#### **Что произойдет, если вызвать open-метод, переопределенный в дочернем классе, внутри блока `init` базового класса?**
* **Ответ:** Произойдет вызов метода дочернего класса на еще *неинициализированном* объекте дочернего класса. В JVM вызовы нестатических методов разрешаются динамически через таблицу виртуальных методов (`vtable`). Поскольку родительский блок `init` выполняется *раньше*, чем инициализируются свойства дочернего класса, переопределенный метод дочернего класса обратится к своим свойствам, которые в этот момент находятся в дефолтном состоянии (для ссылочных типов — `null`). Это приводит к `NullPointerException` при работе со свойствами, объявленными как `val` и помеченными как ненулевые (`non-nullable`).
* **Пример:**
```kotlin
open class BaseForm {
    init {
        // Опасный вызов open-метода из конструктора родителя
        setupView() 
    }
    open fun setupView() {}
}

class RegistrationForm : BaseForm() {
    // Свойство гарантированно не-null на уровне компилятора
    private val passwordInput: String = "RequiredField" 

    override fun setupView() {
        // На момент этого вызова passwordInput еще равен NULL в памяти JVM!
        println("Длина поля: ${passwordInput.length}") // Выбросит NullPointerException
    }
}
```

#### **Можно ли объявить блок `init` во внутриклассовом объекте `companion object`?**
* **Ответ:** Да. Блок `init` внутри `companion object` компилируется в статический блок инициализации класса (метод `<clinit>` в байткоде Java). Он выполняется ровно один раз, когда класс впервые загружается в память загрузчиком классов (`ClassLoader`), перед созданием каких-либо экземпляров класса или обращением к его статическим членам.
* **Пример:** Настройка конфигурации БД один раз при старте работы с классом.
```kotlin
class DatabaseConnector {
    companion object {
        private val connectionPool: List<String>
        
        init {
            println("Статический блок инициализации companion object запущен.")
            connectionPool = listOf("Conn_1", "Conn_2", "Conn_3")
        }
    }
}
```

## 2. Data Class и Enum

### **Базовые вопросы**

#### **Что такое `data class`?**
* **Ответ:** Это класс, предназначенный исключительно для хранения данных (моделей данных). Компилятор Kotlin автоматически генерирует реализации методов `equals()`, `hashCode()`, `toString()`, а также метод `copy()` и набор функций деструктуризации `componentN()` для всех свойств, объявленных в **первичном конструкторе**.
* **Пример:**
```kotlin
data class User(val id: Long, val name: String, val email: String)

val user1 = User(101L, "Ivan", "ivan@mail.ru")
val user2 = user1.copy(email = "new_ivan@mail.ru") // Удобно обновляем только одно свойство
```

---

#### **Почему `data class` должен иметь хотя бы один параметр в первичном конструкторе?**
* **Ответ:** Компилятор использует свойства первичного конструктора как математический базис для генерации методов сравнения и копирования. Без свойств методы `equals()` и `hashCode()` сравнивали бы только ссылки (как обычные объекты), а методы `copy()` и `componentN()` не имели бы сигнатур для генерации, что лишает концепцию `data class` практического смысла.
* **Пример:** Попытка объявить пустой `data class` приведет к ошибке компиляции.
```kotlin
// Ошибка компиляции: Data class must have at least one primary constructor parameter
data class ServerResponse() 
```

---

### **Глубокие вопросы**

#### **Как именно генерируются функции деструктуризации `componentN()` под капотом?**
* **Ответ:** Для каждого параметра в первичном конструкторе компилятор генерирует открытую функцию с префиксом `component` и порядковым номером свойства (начиная с 1). Возвращаемый тип функции строго соответствует типу свойства. При использовании синтаксиса деструктуризации компилятор заменяет эту запись на вызовы соответствующих функций `componentN()`.
* **Пример:**
```kotlin
// Исходный код Kotlin
data class GeoCoordinate(val latitude: Double, val longitude: Double)

val coordinate = GeoCoordinate(55.75, 37.61)
val (lat, lon) = coordinate

// Во что этот код компилируется (упрощенно):
// Компилятор сгенерировал в классе GeoCoordinate методы:
// public final double component1() { return this.latitude; }
// public final double component2() { return this.longitude; }

// Деструктуризация на уровне байткода превращается в:
val lat = coordinate.component1()
val lon = coordinate.component2()
```

#### **Как устроен `enum class` на уровне JVM? Каковы накладные расходы на вызов метода `values()` и как Kotlin 1.9 решает эту проблему?**
* **Ответ:** На уровне JVM `enum class` является обычным классом, расширяющим абстрактный `java.lang.Enum`. Каждая константа перечисления представлена синглтон-объектом, хранящимся в статическом поле класса.
  * **Проблема `values()`:** Традиционный Java/Kotlin метод `MyEnum.values()` при каждом вызове возвращает **новый массив** констант, клонируя внутренний массив. Это сделано для предотвращения модификации исходного массива пользовательским кодом, но приводит к постоянным аллокациям памяти в куче при частых вызовах в циклах.
  * **Решение в Kotlin 1.9+ (`entries`):** Было введено свойство `entries`, которое возвращает неизменяемый список `EnumEntries`, кэшируемый на уровне класса. Оно не выполняет клонирование при каждом обращении, что исключает лишние аллокации.
* **Пример:**
```kotlin
enum class NetworkStatus {
    ONLINE, OFFLINE, CONNECTING;
}

// Оптимальный обход констант без выделения памяти под новые массивы
fun checkStatus() {
    // В Kotlin 1.9+ вместо values() используем entries
    NetworkStatus.entries.forEach { status ->
        println("Статус: ${status.name}")
    }
}
```

### **Каверзные вопросы**

#### **Почему наследование `data class` от другого `data class` запрещено на уровне компилятора?**
* **Ответ:** Наследование нарушило бы математический контракт метода `equals()`, а именно свойство **симметричности** (если `A == B`, то и `B == A`). Если бы дочерний `data class` добавил новое свойство в свой конструктор, сгенерированный родительский `equals()` не знал бы об этом свойстве и сравнивал бы только родительские поля (возвращая `true`). Сгенерированный дочерний `equals()` учитывал бы новое свойство и возвращал бы `false` при сравнении с родителем, что ломает логику работы коллекций и хэш-таблиц JVM.
* **Пример:**
```kotlin
// Данный код запрещен компилятором (data class всегда final)
// open data class User(val id: Long)
// data class Admin(val id: Long, val role: String) : User(id)

// Если бы это было разрешено:
// val u = User(1)
// val a = Admin(1, "SuperAdmin")
// u.equals(a) -> вернет true (сравнивает только id)
// a.equals(u) -> вернет false (так как у u нет поля role)
// Симметрия нарушена: u == a, но a != u. Это недопустимо.
```

#### **Что произойдет, если мы используем `data class` с `var` свойствами в качестве ключа в `HashMap`, а затем изменим одно из этих свойств?**
* **Ответ:** Объект станет недоступным по ключу в `HashMap`, возникнет утечка памяти. Хэш-таблицы распределяют элементы по корзинам (`buckets`) на основе вычисленного `hashCode()` ключа на момент вставки. При изменении `var`-свойства хэш-код объекта пересчитывается. Когда мы вызываем `map.get(key)`, хэш-таблица пытается вычислить индекс корзины по *новому* хэш-коду, обращается не к той корзине, где физически лежит объект, и возвращает `null`. Сам объект при этом остается в мапе навсегда, забирая память.
* **Пример:**
```kotlin
data class CacheKey(var scopeId: String)

val cache = HashMap<CacheKey, String>()
val key = CacheKey("initial_scope")

cache[key] = "Heavy_Cached_UI_Data"

// Изменяем значение свойства, участвующего в вычислении hashCode()
key.scopeId = "new_scope"

// Пытаемся получить данные по тому же самому ключу
val cachedData = cache[key] 

println("Данные из кэша: $cachedData") // Выведет NULL! Объект утерян в памяти HashMap.
```

# Блок 2: Функциональные возможности Kotlin

## 3. Inline, Generics, Abstract Classes

### **Базовые вопросы**

#### **Что такое `inline` функция?**
* **Ответ:** Это функция, для которой компилятор заменяет вызовы на непосредственную вставку тела функции и ее лямбда-параметров в место вызова. Это избавляет от накладных расходов на создание объектов анонимных классов интерфейса `Function` в куче для лямбд и сохраняет плоский стек вызовов.
* **Пример:**
```kotlin
inline fun <T> runWithTransaction(db: SQLiteDatabase, block: () -> T): T {
    db.beginTransaction()
    try {
        val result = block()
        db.setTransactionSuccessful()
        return result
    } finally {
        db.endTransaction()
    }
}
```

---

#### **Что такое ковариантность (`out`) и контравариантность (`in`)?**
* **Ответ:**
  * **Ковариантность (`out T`)** позволяет использовать класс с типом-наследником там, где ожидается базовый тип. Класс может только отдавать (возвращать) тип `T`, но не принимать его на вход.
  * **Контравариантность (`in T`)** позволяет использовать класс с базовым типом там, где ожидается тип-наследник. Класс может только принимать (потреблять) тип `T` в параметрах методов, но не возвращать его.
* **Пример:**
```kotlin
open class Animal
class Dog : Animal()

// out: Иерархия сохраняется. List<Dog> является подтипом List<Animal>
val dogs: List<Dog> = listOf(Dog())
val animals: List<Animal> = dogs // Безопасно, так как List имеет тип out T

// in: Иерархия разворачивается. Consumer<Animal> является подтипом Consumer<Dog>
interface Consumer<in T> { fun consume(item: T) }
val animalConsumer = object : Consumer<Animal> {
    override fun consume(item: Animal) { println("Потребили животное") }
}
val dogConsumer: Consumer<Dog> = animalConsumer // Безопасно передать потребителя всех животных
```

---

### **Глубокие вопросы**

#### **Каков механизм работы `reified` типов и почему они невозможны без `inline`?**
* **Ответ:** В JVM информация об обобщенных типах (Generics) стирается во время компиляции (`Type Erasure`). В рантайме невозможно выполнить проверку `T is String` или получить класс типа через `T::class.java`. 
Но если функция помечена как `inline`, ее тело копируется в место вызова. В месте вызова конкретный тип `T` компилятору уже известен статически (например, `apiService.fetchData<User>()`). Компилятор заменяет абстрактный параметр `T` в инлайненном коде на прямую ссылку на байткод класса `User.class` (инструкция `ldc` в байткоде). Это возможно только при копировании кода (инлайнинге).
* **Пример:** Безопасный парсинг JSON без явной передачи класса.
```kotlin
inline fun <reified T> String.toKotlinObject(): T {
    val serializer = Json.serializersModule.serializer<T>()
    return Json.decodeFromString(serializer, this)
}

// Использование:
val jsonString = """{"id":1,"name":"Alice"}"""
val user = jsonString.toKotlinObject<User>() // Тип User подставляется на этапе компиляции
```

---

#### **Что такое `@PublishedApi` и зачем он нужен в контексте `inline` функций?**
* **Ответ:** `inline` функции копируются в место вызова, поэтому они не могут обращаться к закрытым (`private`) членам класса, так как место вызова может находиться в другой сборке или пакете, где к этим членам нет физического доступа. Чтобы не открывать видимость внутренних членов для всего внешнего кода (сохраняя инкапсуляцию), их помечают как `internal` и добавляют аннотацию `@PublishedApi`. Это дает компилятору право обращаться к ним при генерации инлайненного кода, но скрывает их от автодополнения в IDE для сторонних разработчиков.
* **Пример:**
```kotlin
class ImageDownloader {
    @PublishedApi
    internal val diskCache = LruCache<String, Bitmap>(1024)

    // Функция публична, инлайнится у клиентов
    inline fun cacheImage(url: String, fetcher: () -> Bitmap) {
        val cached = diskCache.get(url) // Доступ разрешен благодаря @PublishedApi
        if (cached == null) {
            diskCache.put(url, fetcher())
        }
    }
}
```

### **Каверзные вопросы**

#### **Можно ли объявить рекурсивную `inline` функцию?**
* **Ответ:** Нет. Рекурсивный инлайнинг привел бы к бесконечному циклу генерации байткода на этапе компиляции, так как компилятор пытался бы бесконечно вставлять тело функции внутрь самой себя. Компилятор Kotlin превентивно выдает ошибку компиляции при обнаружении рекурсии в `inline` методах.
* **Пример:**
```kotlin
// Ошибка компиляции: "Infinite recursion in inline function"
inline fun findDirectories(file: File, action: (File) -> Unit) {
    if (file.isDirectory) {
        file.listFiles()?.forEach { 
            findDirectories(it, action) // Рекурсивный вызов запрещен!
        }
    } else {
        action(file)
    }
}
```

---

#### **Почему нельзя использовать `reified` тип `T` для создания нового экземпляра класса (например, `val t = T()`)?**
* **Ответ:** Несмотря на то, что компилятор знает тип `T` на месте вызова, он не имеет никакой информации о конструкторах этого класса на этапе компиляции самой `inline` функции. Класс `T` может вообще не иметь конструкторов по умолчанию, быть интерфейсом или абстрактным классом. Единственный способ создать экземпляр — использовать рефлексию после проверки наличия конструктора.
* **Пример:**
```kotlin
inline fun <reified T : Any> createNewInstance(): T {
    // val item = T() // Ошибка компиляции: Unresolved reference: T()
    
    // Как сделать правильно через рефлексию с обработкой ошибок:
    return T::class.java.getDeclaredConstructor().newInstance()
}
```

## 4. Interfaces и Sealed Interfaces

### **Базовые вопросы**

#### **Может ли интерфейс хранить состояние в Kotlin?**
* **Ответ:** Нет, интерфейсы не могут хранить состояние. Они компилируются в JVM-интерфейсы, которые физически не могут иметь переменных экземпляра (`fields`). Kotlin позволяет объявлять свойства в интерфейсах, но они не генерируют `backing field`. Эти свойства транслируются в абстрактные методы-геттеры и сеттеры, которые обязаны реализовывать классы-наследники, либо предоставляют логику вычисления на лету через кастомные геттеры.
* **Пример:**
```kotlin
interface TokenProvider {
    val rawToken: String // Абстрактный геттер, класс обязан переопределить и предоставить поле
    
    val isExpired: Boolean
        get() = rawToken.startsWith("EXPIRED_") // Кастомный геттер, состояние вычисляется динамически
}
```

---

#### **Что такое `sealed interface`?**
* **Ответ:** Это интерфейс, закрытый для реализации за пределами пакета/модуля, в котором он объявлен. Компилятор строго отслеживает всех наследников на этапе сборки, что позволяет использовать безопасную и исчерпывающую проверку ветвей `when` без необходимости указывать ветку `else`.
* **Пример:**
```kotlin
sealed interface UserActionEvent {
    object Logout : UserActionEvent
    data class SendMessage(val text: String) : UserActionEvent
}

fun handleAction(event: UserActionEvent) {
    // Ветка else не требуется, так как иерархия закрыта
    when (event) {
        UserActionEvent.Logout -> println("Выход")
        is UserActionEvent.SendMessage -> println("Текст: ${event.text}")
    }
}
```

### **Глубокие вопросы**

#### **Как Kotlin реализует default-методы интерфейсов на старых версиях Android SDK (ниже API 24), где JVM не поддерживала их нативно?**
* **Ответ:** До Java 8 (API 24 в Android) интерфейсы не могли иметь реализации методов. Для поддержки совместимости компилятор Kotlin генерирует для каждого интерфейса с реализованными методами вспомогательный внутренний класс `MyInterface$DefaultImpls`. Любая дефолтная реализация компилируется как статический метод этого класса-помощника, принимающий экземпляр интерфейса в качестве первого аргумента (наподобие ссылки `this`). Класс, реализующий интерфейс, под капотом переопределяет метод и перенаправляет вызов в статический метод вспомогательного класса.
* **Пример:**
```kotlin
// Исходный код Kotlin
interface FeatureToggle {
    fun isEnabled(): Boolean = true
}

// Что генерирует компилятор для совместимости со старыми JVM (упрощенно):
public interface FeatureToggle {
   boolean isEnabled();

   // Статический хелпер
   public static final class DefaultImpls {
      public static boolean isEnabled(FeatureToggle $this) {
         return true;
      }
   }
}
```

#### **В чем разница между `sealed class` и `sealed interface` с точки зрения генерации метаданных?**
* **Ответ:** 
  * На уровне JVM (начиная с Java 15+) оба типа транслируются в нативные запечатанные классы/интерфейсы JVM с атрибутом `PermittedSubclasses` в байткоде.
  * На уровне архитектуры Kotlin ключевое различие в наследовании: класс может реализовывать несколько `sealed interface`, но наследоваться может только от одного `sealed class`. Это позволяет обходить ограничение одиночного наследования классов JVM при проектировании закрытых иерархий.
* **Пример:**
```kotlin
sealed interface Identifiable { val id: String }
sealed interface Auditable { val updatedAt: Long }

// Класс успешно входит в обе закрытые иерархии одновременно
class SystemLog(
    override val id: String, 
    override val updatedAt: Long
) : Identifiable, Auditable
```

### **Каверзные вопросы**

#### **Что произойдет при конфликте default-методов в интерфейсах?**
* **Ответ:** Компилятор Kotlin выдаст ошибку сборки на этапе анализа связей классов. Он потребует от класса-наследника явно переопределить конфликтующий метод, чтобы разрешить неоднозначность. Внутри переопределенного метода разработчик может явно указать, метод какого родительского интерфейса должен быть вызван с помощью ключевого слова `super` с указанием интерфейса в угловых скобках: `super<InterfaceName>.method()`.
* **Пример:**
```kotlin
interface CardReader {
    fun read() { println("Читаем карту") }
}

interface NfcReader {
    fun read() { println("Читаем NFC") }
}

// Ошибка компиляции без переопределения метода read()
class PosTerminal : CardReader, NfcReader {
    override fun read() {
        // Разрешаем конфликт явно
        super<CardReader>.read() // Вызовет логику CardReader
        super<NfcReader>.read()  // Вызовет логику NfcReader
    }
}
```

#### **Можно ли объявить `sealed interface`, у которого вообще нет наследников в текущем модуле?**
* **Ответ:** Да, синтаксически это разрешено. Однако такой интерфейс невозможно будет реализовать ни в каком другом модуле, так как все наследники `sealed` типов обязаны находиться в том же модуле и том же пакете. Это превращает такой интерфейс в "изолированную систему", которую невозможно расширить извне.
* **Пример:**
```kotlin
// Модуль ":core"
sealed interface CorePlugin // Нет наследников в ":core"

// Модуль ":feature" (зависит от ":core")
// Ошибка компиляции: "Inheritor of a sealed class or interface declared in another module"
class CustomPlugin : CorePlugin 
```

# Блок 3: Корутины (Coroutines)

## 5. Основы корутин

### **Базовые вопросы**

#### **Что такое Coroutine?**
* **Ответ:** Это легковесный поток управления, выполняющийся поверх пула реальных потоков ОС. Ключевая особенность корутины — поддержка неблокирующей приостановки работы (`suspension`). Когда корутина приостанавливает выполнение (например, ожидает ответ от сервера), она освобождает физический поток, на котором выполнялась. Этот поток может быть задействован ОС для выполнения других корутин или задач UI.
* **Пример:**
```kotlin
// Выполняется на главном потоке без фризов UI
viewModelScope.launch(Dispatchers.Main) {
    val data = withContext(Dispatchers.IO) {
        api.downloadData() // Поток Main свободен для отрисовки UI во время скачивания
    }
    updateCards(data)
}
```

---

#### **Что делает `suspend`?**
* **Ответ:** Это маркер для компилятора, указывающий, что функция может приостановить выполнение корутины. Такие функции могут вызывать другие приостанавливающие функции и изменять состояние корутины. Они не блокируют поток, на котором вызваны, но требуют вызова из другой `suspend` функции или из строителя корутин (`CoroutineScope`).
* **Пример:**
```kotlin
suspend fun getUserProfile(userId: String): Profile {
    val id = fetchUserId() // Точка приостановки
    return apiService.getProfile(id) // Точка приостановки
}
```

---

### **Глубокие вопросы**

#### **Как устроен механизм приостановки (suspension) на уровне стейт-машины JVM?**
* **Ответ:** Компилятор Kotlin переписывает сигнатуру каждой `suspend` функции на этапе компиляции, используя паттерн **CPS (Continuation Passing Style)**. 
  1. В конец параметров метода добавляется скрытый параметр `Continuation<Any?>` (колбэк состояния корутины). Возвращаемый тип функции меняется на `Any?`.
  2. Тело функции преобразуется в класс стейт-машины (наследник `CoroutineImpl`), содержащий переменную состояния `label`.
  3. Код функции разбивается на блоки (состояния) по границам вызовов других `suspend` функций.
  4. При вызове функция проверяет `label`. Перед каждым асинхронным вызовом `label` увеличивается. Если метод возвращает маркер `COROUTINE_SUSPENDED`, функция завершает работу на текущем потоке.
  5. Когда асинхронный процесс завершается, у переданного `Continuation` вызывается метод `resumeWith()`, стейт-машина запускается заново, переходя по `switch(label)` к следующему состоянию.
* **Пример:** Как компилятор преобразует `suspend` функцию в стейт-машину на Java (псевдокод):
```kotlin
// Оригинальный код Kotlin
suspend fun fetchSession(): String {
    val token = api.getToken() // Точка приостановки
    return "Session_$token"
}

// Во что превращается на уровне байткода JVM (упрощенный псевдокод):
public Object fetchSession(Continuation completion) {
    // Внутренний класс стейт-машины для сохранения переменных между приостановками
    class FetchSessionSM extends CoroutineImpl {
        Object result;
        int label = 0;
        
        @Override
        public Object invokeSuspend(Object result) {
            this.result = result;
            this.label |= Integer.MIN_VALUE;
            return fetchSession(this);
        }
    }

    FetchSessionSM sm = (completion instanceof FetchSessionSM) 
        ? (FetchSessionSM) completion 
        : new FetchSessionSM(completion);

    switch (sm.label) {
        case 0: // Начальный запуск
            sm.label = 1;
            Object res = api.getToken(sm); // Передаем стейт-машину как коллбэк
            if (res == COROUTINE_SUSPENDED) return COROUTINE_SUSPENDED; // Уходим с потока
        case 1: // Восстановление после получения токена
            String token = (String) sm.result;
            return "Session_" + token;
        default:
            throw new IllegalStateException("Wrong state label");
    }
}
```

---

#### **Как диспетчер `Dispatchers.IO` разделяет потоки с `Dispatchers.Default` под капотом?**
* **Ответ:** Оба диспетчера используют единый общий пул потоков, управляемый планировщиком `CoroutineScheduler`. Разделение происходит на уровне логического лимитирования потоков (через класс `LimitingDispatcher`).
  * `Dispatchers.Default` ограничен числом ядер процессора (но не менее 2). Он рассчитан на непрерывную загрузку CPU вычислительными задачами.
  * `Dispatchers.IO` имеет динамический лимит в 64 потока (или равен числу ядер, если оно больше). Он спроектирован для блокирующих операций (диск, сеть), где потоки часто простаивают в ожидании ввода-вывода.
  При переключении с `Default` на `IO` корутина может продолжить выполняться на том же самом физическом потоке, если планировщик видит свободные лимиты, что минимизирует расходы на реальное переключение контекста процессора.
* **Пример:**
```kotlin
viewModelScope.launch(Dispatchers.Default) {
    println("Default thread: ${Thread.currentThread().name}") 
    // Напечатает: DefaultDispatcher-worker-1
    
    withContext(Dispatchers.IO) {
        println("IO thread: ${Thread.currentThread().name}") 
        // Напечатает тот же поток DefaultDispatcher-worker-1, если лимиты позволяют!
    }
}
```

### **Каверзные вопросы**

#### **Что произойдет, если запустить 1000 корутин на `Dispatchers.Default` и в каждой из них вызвать блокирующий `Thread.sleep(1000)`?**
* **Ответ:** Произойдет полная блокировка пула потоков `Dispatchers.Default` (Thread Starvation). `Thread.sleep()` блокирует физический поток ОС, не позволяя планировщику корутин задействовать его для выполнения других задач. Поскольку пул `Default` ограничен количеством ядер (например, 8 ядер = 8 потоков), первые 8 корутин захватят все 8 потоков и уснут. Остальные 992 корутины останутся стоять в очереди без движения. Любая другая работа в приложении на `Dispatchers.Default` (включая рекомпозицию UI или вычисления) зависнет до завершения сна первых потоков.
* **Пример:**
```kotlin
fun startHeavyCalculations() {
    repeat(100) { index ->
        // Ломаем планировщик блокирующим сном
        scope.launch(Dispatchers.Default) {
            Thread.sleep(1000) 
            println("Задача $index завершена")
        }
    }
    
    // Данная корутина зависнет и не сможет начать выполнение, пока не освободится поток
    scope.launch(Dispatchers.Default) {
        println("Я хочу выполниться немедленно!")
    }
}
```

#### **В чем разница между `coroutineContext[Job]` и текущим контекстом внутри блока `withContext`?**
* **Ответ:** Функция `withContext` под капотом создает специальную корутину-обертку (`DispatchedCoroutine`), которая заменяет текущий `Job` в контексте на свой собственный дочерний `Job`. Это необходимо для того, чтобы `withContext` мог отслеживать отмену запущенного внутри него блока и корректно приостанавливать вызывающую корутину до завершения работы блока. Соответственно, объект `Job`, полученный через `coroutineContext[Job]` внутри `withContext`, физически будет являться другим экземпляром класса, нежели `Job` внешней корутины.
* **Пример:**
```kotlin
scope.launch(Dispatchers.Main) {
    val outerJob = coroutineContext[Job]
    
    withContext(Dispatchers.IO) {
        val innerJob = coroutineContext[Job]
        println("Джобы идентичны? ${outerJob === innerJob}") // Напечатает: FALSE
    }
}
```

## 6. Flow

### **Базовые вопросы**

#### **Что такое Flow? Почему он является "холодным" (cold) потоком данных?**
* **Ответ:** `Flow` — это асинхронный поток данных, последовательно поставляющий значения. Он является **холодным**, потому что сам блок кода внутри конструктора `flow { ... }` не начинает выполняться, пока на этом потоке не будет вызван терминальный оператор (самый частый — `collect()`). У `Flow` нет собственного состояния и активных подписчиков; каждый вызов `collect` создает новую изолированную цепочку выполнения с самого начала.
* **Пример:**
```kotlin
val usersFlow = flow {
    println("Продюсер запущен") // Напечатается только при вызове collect()
    val users = api.getUsers()
    emit(users)
}

// Запуск сбора данных
viewModelScope.launch {
    usersFlow.collect { println(it) } // Только сейчас пойдет запрос в сеть
}
```

---

### **Глубокие вопросы**

#### **Как реализован механизм Backpressure в Kotlin Flow на уровне корутин?**
* **Ответ:** В отличие от RxJava, где для управления переполнением буфера используются сложные программные стратегии и запросы емкости, в Kotlin Flow эта проблема решена на уровне архитектуры корутин с помощью приостановки (`suspension`).
  Интерфейс `FlowCollector` объявляет метод `emit` как `suspend` функцию: `suspend fun emit(value: T)`.
  Если коллектор медленно обрабатывает данные (например, делает запись в базу данных), выполнение функции `collect` приостанавливается. Соответственно, приостанавливается и вызвавшая его функция `emit` в продюсере. Продюсер физически не сможет отправить следующий элемент (перейти к следующей строке кода), пока корутина коллектора не возобновит свою работу после обработки текущего элемента.
* **Пример:**
```kotlin
flow {
    println("Продюсер: Отправляем 1")
    emit(1) // Приостановится здесь, пока коллектор не закончит обработку
    println("Продюсер: Отправляем 2")
    emit(2)
}.collect { value ->
    println("Коллектор получил: $value")
    delay(2000) // Медленная обработка элемента
    println("Коллектор закончил обработку: $value")
}
```

---

#### **Что такое `SharedFlow` с параметром `replay = 0` и `extraBufferCapacity = 0` при медленном подписчике? Какая стратегия буферизации сработает по умолчанию?**
* **Ответ:** По умолчанию для `SharedFlow` применяется стратегия `BufferOverflow.SUSPEND`.
  Если у нас есть медленный подписчик, буфер которого переполнен (при `replay = 0` и `extraBufferCapacity = 0` любое новое значение переполняет буфер немедленно, если подписчик занят), то корутина, вызывающая `emit()`, будет приостановлена (`suspended`). Отправитель будет заблокирован в ожидании, пока медленный подписчик не освободится и не заберет отправленное значение. Если мы хотим избежать приостановки отправителя, нужно использовать метод `tryEmit()` или изменить стратегию переполнения буфера (например, на `DROP_OLDEST` или `DROP_LATEST`).
* **Пример:**
```kotlin
val eventStream = MutableSharedFlow<UiEvent>(
    replay = 0,
    extraBufferCapacity = 0,
    onBufferOverflow = BufferOverflow.SUSPEND // Отправитель ждет медленного UI
)
```

### **Каверзные вопросы**

#### **Будет ли перехвачено исключение оператором `catch` в следующем коде?**
```kotlin
flow { emit(1) }
    .catch { emit(-1) }
    .collect { throw RuntimeException("Error") }
```
* **Ответ:** **Нет, не будет.** Исключение приведет к крашу приложения.
  Оператор `catch` во `Flow` перехватывает исключения, которые возникают исключительно **выше по течению (upstream)** относительно самого оператора `catch`. Ошибки, возникшие ниже по течению (в терминальном операторе `collect` или в операторах после `catch`), им не обрабатываются. Это фундаментальное правило обеспечения изоляции ошибок во `Flow`.
* **Пример исправления (обработка всех ошибок всей цепочки):**
```kotlin
// ПЛОХО: Краш
flow { emit(1) }.catch { println("Поймали") }.collect { error() }

// ПРАВИЛЬНО: Переносим логику обработки в OnEach до вызова catch
flow { emit(1) }
    .onEach { value ->
        throw RuntimeException("Ошибка в обработчике")
    }
    .catch { e ->
        println("Успешно поймали ошибку: ${e.message}") // Ошибка будет перехвачена!
    }
    .collect() // Пустой коллектор
```

---

#### **Что произойдет, если вызвать `emit()` из другого потока (например, внутри `flow { ... }` запустить `thread { emit(x) }`)?**
* **Ответ:** Произойдет краш с исключением `IllegalStateException` с сообщением о нарушении сохранности контекста (`Flow invariant violation`).
  `Flow` гарантирует, что эмит данных происходит строго в том же контексте корутины (и на том же потоке), на котором был запущен сборщик (`collect`). Запуск прямого вызова `emit` из стороннего Java-потока или другой корутины ломает стейт-машину выполнения. Для многопоточной отправки данных необходимо использовать специализированный билдер `callbackFlow` или `channelFlow`, который под капотом использует потокобезопасные каналы (`Channels`).
* **Пример ошибки и её исправления:**
```kotlin
// ОШИБКА: Краш приложения!
val badFlow = flow {
    Thread { emit("Data") }.start() // Ошибка нарушения контекста!
}

// ПРАВИЛЬНО: Работа с коллбэками внешних SDK
val safeFlow = callbackFlow {
    val listener = object : SdkListener {
        override fun onData(data: String) {
            trySend(data) // Безопасная отправка из любого потока
        }
    }
    sdk.register(listener)
    awaitClose { sdk.unregister(listener) } // Очистка при отмене подписки
}
```

## 7. Clean Architecture, MVVM, MVI и концепция SSOT

### **Базовые вопросы**

#### **Что такое SSOT?**
* **Ответ:** **Single Source of Truth (Единый источник истины)** — это архитектурный паттерн проектирования, согласно которому определенные данные (например, профиль пользователя, список заказов, статус сессии) имеют ровно одно место хранения и изменения. Никакие другие компоненты приложения не имеют права хранить дубликаты этих данных или изменять их в обход этого источника. Все потребители подписываются на изменения этого источника и получают данные в реактивном режиме (только для чтения).
* **Пример:** Локальная база данных Room выступает как SSOT для отображения списка товаров.
```kotlin
class ProductRepository(
    private val productDao: ProductDao,
    private val api: ProductApi
) {
    // UI подписывается на БД. Все изменения на сервере транслируются сначала в БД.
    fun observeProducts(): Flow<List<ProductEntity>> = productDao.getProductsFlow()

    // Сетевой запрос обновляет только БД, UI реагирует на это автоматически
    suspend fun refreshProducts() {
        val remoteProducts = api.fetchProducts()
        productDao.insertAll(remoteProducts.map { it.toEntity() })
    }
}
```

---

#### **В чем разница между MVVM и MVI?**
* **Ответ:**
  * В **MVVM** ViewModel содержит несколько независимых наблюдаемых источников состояния (например, `LiveData` или `StateFlow` для списка товаров, статуса загрузки, ошибок). View подписывается на каждый поток отдельно. Связь односторонняя, но потоков данных много.
  * В **MVI** все состояние экрана упаковано в единый неизменяемый объект `UiState` (State). Пользовательские действия транслируются во ViewModel в виде потока намерений (`Intent`). ViewModel обрабатывает входящий `Intent`, генерирует новый `UiState` и отправляет его во View. Поток данных строго однонаправлен (**Unidirectional Data Flow — UDF**), а состояние экрана является атомарным.
* **Пример:**
```kotlin
// Атомарное состояние в MVI
data class ProfileUiState(
    val isLoading: Boolean = false,
    val profile: Profile? = null,
    val errorMessage: String? = null
)

sealed interface ProfileIntent {
    object LoadProfile : ProfileIntent
    data class UpdateBio(val newBio: String) : ProfileIntent
}
```

### **Глубокие вопросы**

#### **Как организовать маппинг моделей между слоями в Clean Architecture, чтобы избежать дублирования кода, но сохранить независимость?**
* **Ответ:** Необходимо строго разделять сущности по слоям ответственности:
  1. **Data Layer:** `UserDto` (сетевая модель Retrofit/Ktor), `UserEntity` (таблица Room).
  2. **Domain Layer:** `User` (чистый Kotlin-класс бизнес-логики без аннотаций фреймворков).
  3. **Presentation Layer:** `UserUiModel` (модель, оптимизированная для вывода в UI, например, со склеенным именем и отформатированной датой).
  Маппинг (преобразование) должен происходить на стыке слоев. Репозиторий в Data Layer принимает `UserDto` из сети, маппит его в `User` и отдает в UseCase. ViewModel принимает `User` из UseCase, маппит его в `UserUiModel` и отдает в UI. Это предотвращает распространение изменений внешних API на бизнес-логику.
* **Пример:**
```kotlin
// Data Layer
@Serializable
data class UserNetworkDto(val first_name: String, val last_name: String)

fun UserNetworkDto.toDomain(): UserDomainModel {
    return UserDomainModel(name = "$first_name $last_name")
}

// Domain Layer
data class UserDomainModel(val name: String)

// Presentation Layer
data class UserUiState(val displayName: String)

fun UserDomainModel.toUiState(): UserUiState {
    return UserUiState(displayName = "Пользователь: $name")
}
```

#### **Как устроен жизненный цикл ViewModel и за счет чего она выживает при повороте экрана?**
* **Ответ:** При смене конфигурации (поворот экрана) Activity уничтожается и создается заново. Однако объект `ViewModelStore` (хранилище всех ViewModel) не уничтожается.
  В процессе уничтожения Activity система вызывает метод `retainNonConfigurationInstances()`, упаковывая `ViewModelStore` во внутреннюю структуру. При создании нового инстанса Activity система возвращает сохраненный `ViewModelStore` через метод `getLastNonConfigurationInstance()`. Сама `ViewModel` очищается (вызывается метод `onCleared()`) только тогда, когда Activity закрывается окончательно (например, при нажатии кнопки «Назад» или вызове `finish()`).
* **Пример:** Системный API androidx (схематичная логика выживания):
```kotlin
// Упрощенный код внутри androidx ComponentActivity
override fun onRetainCustomNonConfigurationInstance(): Any? {
    // Сохраняем хранилище ViewModel при смене конфигурации
    return viewModelStore 
}

// При создании новой Activity хранилище восстанавливается
val retainedStore = lastCustomNonConfigurationInstance as? ViewModelStore
if (retainedStore != null) {
    this.viewModelStore = retainedStore
}
```

### **Каверзные вопросы**

#### **Правда ли, что использование `AndroidViewModel` (с передачей `Application` контекста) полностью защищает от утечек памяти?**
* **Ответ:** Нет, это миф. Использование `Application` контекста лишь защищает от утечки короткоживущего контекста конкретной `Activity`. Но если `AndroidViewModel` зарегистрирует слушатель в синглтоне (который живет на протяжении работы всего приложения) и передаст туда коллбэк, содержащий неявную ссылку на саму `ViewModel`, то эта `ViewModel` утечет. Сборщик мусора не сможет удалить ее из памяти после закрытия экрана.
* **Пример утечки:**
```kotlin
class LocationTracker @Inject constructor(val context: Context) {
    var onLocationChanged: ((Location) -> Unit)? = null // Слушатель в синглтоне
}

class BadViewModel(
    application: Application, 
    private val tracker: LocationTracker // Системный синглтон
) : AndroidViewModel(application) {
    
    init {
        // УТЕЧКА ПАМЯТИ: tracker хранит ссылку на лямбду, которая неявно держит ссылку 
        // на экземпляр BadViewModel. При закрытии экрана ViewModel останется в памяти.
        tracker.onLocationChanged = { location ->
            updateLocationInUi(location)
        }
    }
}
```

#### **В MVI архитектуре, если мы отправляем событие навигации как часть единого `UiState` (например, `state.copy(navigateToProfile = true)`), с какой проблемой мы столкнемся при повороте экрана?**
* **Ответ:** Мы столкнемся с проблемой **повторного выполнения одноразового события (Event Replay)**. При повороте экрана новая Activity подписывается на изменения `UiState` во ViewModel, считывает последнее валидное состояние (в котором поле `navigateToProfile` все еще равно `true`) и повторно открывает экран профиля, уходя в бесконечный цикл навигации.
  * **Решение:** Одноразовые события (навигация, тосты, диалоги) должны передаваться независимо от постоянного состояния экрана (`UiState`) через событийные шины — `Channel` или `SharedFlow` со стратегией `replay = 0`.
* **Пример:**
```kotlin
// Одноразовые эффекты экрана
sealed interface ProfileEffect {
    object NavigateToProfile : ProfileEffect
}

class ProfileMviViewModel : ViewModel() {
    // Используем Channel для отправки одноразовых эффектов
    private val _effects = Channel<ProfileEffect>(Channel.BUFFERED)
    val effects = _effects.receiveAsFlow()

    fun onProfileClicked() {
        viewModelScope.launch {
            _effects.send(ProfileEffect.NavigateToProfile) // Сборщик считает и очистит из буфера
        }
    }
}
```

## 8. DI (Dependency Injection)

### **Базовые вопросы**

#### **Что такое Dependency Injection?**
* **Ответ:** Это паттерн проектирования, при котором объект не создает свои зависимости (другие объекты, необходимые ему для работы) самостоятельно, а получает их извне от внешнего управляющего компонента (DI-контейнера). Это снижает связанность кода (`low coupling`), упрощает тестирование и облегчает масштабирование приложения.
* **Пример:**
```kotlin
// Без DI (Жесткая связанность)
class UserService {
    private val api = RetrofitUserApi() // Плохо: класс сам создает зависимость
}

// С DI (Слабая связанность)
class UserService(private val api: UserApi) // Хорошо: зависимость внедряется извне
```

---

#### **Чем отличается Hilt от Dagger?**
* **Ответ:** 
  * **Dagger 2** — это мощная, но сложная библиотека для генерации графа зависимостей на этапе компиляции. Она требует ручного описания компонентов (`Components`), сабкомпонентов (`Subcomponents`), скоупов и ручной привязки к жизненному циклу Android (Activity, Fragment).
  * **Hilt** — это официальная библиотека-обертка над Dagger 2 от Google. Она стандартизирует использование Dagger в Android. Hilt автоматически создает предопределенные компоненты для классов Android (`Application`, `Activity`, `ViewModel`, `Fragment`), управляет их жизненным циклом и предоставляет простые аннотации (например, `@AndroidEntryPoint`), минимизируя шаблонный код (`boilerplate`).

---

### **Глубокие вопросы**

#### **Почему compile-time DI быстрее?**
* **Ответ:** Фреймворки типа Dagger/Hilt выполняют всю тяжелую работу по анализу зависимостей на этапе сборки проекта с помощью процессоров аннотаций (KAPT/KSP). Они генерируют обычные Java-классы фабрик для создания объектов. 
  В рантайме внедрение зависимостей происходит путем выполнения прямого сгенерированного кода (нативных вызовов `new MyService(new MyRepository())`). 
  Рантайм-фреймворки (типа Koin в старых версиях) вынуждены искать соответствия типов в хэш-картах во время выполнения приложения, используя рефлексию, что создает нагрузку на CPU на старте приложения и замедляет холодный запуск (`Cold Start`).

---

#### **Koin раньше был только runtime DI. А сейчас? Есть ли способ проверять зависимости на этапе компиляции?**
* **Ответ:** Да. В современных версиях Koin (3.x–4.x) появились инструменты для проверки графа:
  1. **Koin Annotations + KSP:** Генерация модулей на этапе компиляции. С флагом `KOIN_CONFIG_CHECK=true` можно запустить генератор для частичной проверки графа.
  2. **Koin Compiler Plugin:** Полноценный плагин компилятора Kotlin, который анализирует конфигурацию Koin и может проводить компиляционные проверки типов при использовании специального DSL (пакет `org.koin.plugin.module.dsl.*`).
  3. **Юнит-тестирование графа (`koin-test`):** Стандартный и надежный способ — написание простого теста с вызовом метода `checkModules()`, который проверяет граф зависимостей в тестовой JVM-среде.
* **Пример:**
```kotlin
class KoinDependencyTest : KoinTest {
    @Test
    fun verifyKoinGraph() {
        // Тест падает, если какая-то зависимость отсутствует в графе
        koinApplication {
            modules(appModule, apiModule, databaseModule)
        }.checkModules() 
    }
}
```

### **Каверзные вопросы**

#### **Как неправильный scope может привести к утечке?**
* **Ответ:** Если долгоживущий объект (например, помеченный аннотацией `@Singleton`) запрашивает в качестве зависимости объект с меньшим временем жизни (например, помеченный как `@ActivityScoped`), то синглтон будет удерживать ссылку на этот объект вечно. В результате Activity, которая должна была уничтожиться при закрытии экрана, останется в оперативной памяти, так как на нее прямо или косвенно через зависимости ссылается живой синглтон.
* **Пример утечки:**
```kotlin
@ActivityScoped
class ActivityTracker @Inject constructor(
    @ActivityContext val context: Context // Ссылка на текущую Activity
)

@Singleton // Долгоживущий объект
class AnalyticsService @Inject constructor(
    // КРИТИЧЕСКАЯ УТЕЧКА: Синглтон сохранит ссылку на ActivityTracker, 
    // предотвращая очистку Activity после закрытия экрана.
    private val activityTracker: ActivityTracker 
)
```

## 9. Room

### **Базовые вопросы**

#### **Что такое Room?**
* **Ответ:** Это библиотека-надстройка (ORM) над стандартной базой данных SQLite от Google. Она предоставляет строго типизированный API, проверяет синтаксис SQL-запросов (аннотации `@Query`) на этапе компиляции проекта, генерирует код миграций и интегрируется с реактивными потоками (Flow, LiveData) и корутинами.
* **Пример:**
```kotlin
@Database(entities = [UserEntity::class], version = 1)
abstract class AppDatabase : RoomDatabase() {
    abstract fun userDao(): UserDao
}
```

---

#### **Что такое DAO?**
* **Ответ:** **DAO (Data Access Object)** — это интерфейс или абстрактный класс, определяющий методы для взаимодействия с таблицами базы данных. Room автоматически генерирует реализацию этих методов на этапе компиляции.
* **Пример:**
```kotlin
@Dao
interface UserDao {
    @Insert(onConflict = OnConflictStrategy.REPLACE)
    suspend fun insertUser(user: UserEntity)

    @Query("SELECT * FROM users WHERE userId = :id")
    suspend fun getUserById(id: String): UserEntity?
}
```

---

### **Глубокие вопросы**

#### **Что такое N+1 problem в Room и как ее избежать?**
* **Ответ:** Это проблема производительности, когда для получения списка записей со связанными данными Room сначала делает 1 запрос для получения родительских записей, а затем в цикле выполняет еще N запросов для получения дочерних записей каждой сущности. Это создает огромную нагрузку на диск.
  * **Решение:** Использование аннотации `@Relation` внутри POJO-класса связи совместно с аннотацией `@Transaction` над методом DAO. В этом случае Room сделает ровно два запроса (один для авторов, второй для книг по списку идентификаторов авторов через оператор `IN`) и сам свяжет их в памяти.
* **Пример решения:**
```kotlin
data class UserWithOrders(
    @Embedded val user: UserEntity,
    @Relation(
        parentColumn = "userId",
        entityColumn = "customerOwnerId"
    )
    val orders: List<OrderEntity>
)

@Dao
interface UserDao {
    @Transaction // Обеспечивает выполнение двух запросов атомарно в рамках одной транзакции
    @Query("SELECT * FROM users")
    fun getAllUsersWithOrders(): List<UserWithOrders>
}
```

---

### **Каверзные вопросы**

#### **В чем опасность N+1 проблемы в Room с `@Relation`? Что будет, если забыть аннотацию `@Transaction`?**
* **Ответ:** Если над методом получения связей `@Relation` забыть указать аннотацию `@Transaction`, возникнут две критические проблемы:
  1. **Потеря атомарности (данные разъедутся):** Room выполняет запросы для получения родительских и дочерних записей последовательно. Если в промежутке между первым (родители) и вторым (дети) запросом фоновый поток удалит или изменит данные дочерней таблицы, итоговый собранный объект будет содержать несогласованные или пустые данные.
  2. **Снижение производительности:** Каждый запрос будет выполняться вне транзакции, заставляя SQLite дергать диск для фиксации состояния несколько раз. `@Transaction` гарантирует выполнение всех запросов в рамках единой транзакции SQLite.
* **Пример:**
```kotlin
// ОПАСНО: нет @Transaction! Данные могут стать неконсистентными в рантайме.
@Query("SELECT * FROM users")
fun badGetUsers(): List<UserWithOrders>
```

## 10. Retrofit

### **Базовые вопросы**

#### **Что такое Retrofit?**
* **Ответ:** Это типобезопасный HTTP-клиент для Android и Java от Square. Он преобразует Java/Kotlin интерфейсы с декларативными аннотациями (`@GET`, `@POST`, `@Path`, `@Body`) в готовые HTTP-запросы к веб-службам, делегируя низкоуровневую работу клиенту OkHttp.
* **Пример:**
```kotlin
interface UserApi {
    @GET("users/{id}")
    suspend fun getUser(@Path("id") id: String): UserResponse
}
```

---

#### **Чем отличается от OkHttp?**
* **Ответ:** 
  * **OkHttp** — это низкоуровневая библиотека для отправки HTTP-запросов. Она отвечает за сетевые сокеты, пулы соединений, кэширование, перехватчики сигналов (`Interceptors`), повторные попытки при сбоях и парсинг заголовков.
  * **Retrofit** — это высокоуровневый декларативный фасад над OkHttp. Он избавляет от написания шаблонного кода создания запросов и сериализации ответов (автоматически конвертирует JSON/XML в объекты Kotlin с помощью фабрик типа `MoshiConverterFactory` или `KotlinxSerializationConverterFactory`).

---

### **Глубокие вопросы**

#### **Как Retrofit интегрируется с Coroutines? Как под капотом обрабатывается `suspend`?**
* **Ответ:** Когда метод интерфейса объявлен как `suspend`, Retrofit под капотом использует кастомный `CallAdapter` для корутин. 
  При вызове метода он преобразует стандартный OkHttp вызов в асинхронный (`Call.enqueue()`). Корутина, вызвавшая этот метод, приостанавливается (`suspends`). Retrofit регистрирует коллбэк ответа. При успешном получении ответа или ошибке Retrofit возобновляет работу корутины через вызов `continuation.resume()` или `continuation.resumeWithException()`, делая весь асинхронный процесс линейным для разработчика.
* **Пример:**
```kotlin
// Под капотом Retrofit оборачивает сетевой вызов в suspendCancellableCoroutine:
suspend fun <T> Call<T>.await(): T = suspendCancellableCoroutine { continuation ->
    enqueue(object : Callback<T> {
        override fun onResponse(call: Call<T>, response: Response<T>) {
            if (response.isSuccessful) continuation.resume(response.body()!!)
            else continuation.resumeWithException(HttpException(response))
        }
        override fun onFailure(call: Call<T>, t: Throwable) {
            continuation.resumeWithException(t)
        }
    })
}
```

---

### **Каверзные вопросы**

#### **Что произойдет, если в `suspend` функции Retrofit в качестве возвращаемого типа указать `Response<ResponseBody>`, выполнить запрос, но забыть закрыть тело ответа `response.body()?.close()`?**
* **Ответ:** Произойдет утечка сокетов и исчерпание пула соединений OkHttp (Connection Pool Leak).
  Класс `ResponseBody` удерживает открытый поток (InputStream) к сокету операционной системы. Пока мы явно не закроем его с помощью `.close()` (или полностью не вычитаем данные, что закроет его неявно), OkHttp считает соединение активным и не может вернуть этот сокет обратно в пул для переиспользования другими запросами. Если делать такие запросы регулярно, лимит открытых сокетов исчерпается, и последующие сетевые запросы будут зависать в ожидании свободного соединения.
* **Пример:**
```kotlin
interface FileApi {
    @GET("download/zip")
    suspend fun downloadLargeFile(): Response<ResponseBody>
}

// Вызов:
val response = api.downloadLargeFile()
if (response.isSuccessful) {
    val bytes = response.body()?.bytes() // Данные прочитаны, но поток открыт!
    // КРИТИЧЕСКИ: Необходимо обязательно вызвать response.body()?.close() 
    // или использовать блок 'use': response.body()?.use { ... }
}
```

## 11. ООП и SOLID

### **Базовые вопросы**

#### **Назовите основные принципы ООП.**
* **Ответ:**
  * **Инкапсуляция:** Сокрытие внутренней реализации и данных объекта, доступ к ним предоставляется только через строго контролируемый публичный интерфейс (методы).
  * **Наследование:** Возможность создания нового класса на основе существующего для переиспользования свойств и методов.
  * **Полиморфизм:** Способность объектов с одинаковым интерфейсом иметь разную внутреннюю реализацию методов (динамический полиморфизм через переопределение методов `@Override`).
  * **Абстракция:** Выделение ключевых значимых характеристик объекта предметной области с игнорированием несущественных деталей.

---

#### **Что такое SOLID?**
* **Ответ:** Пять базовых принципов объектно-ориентированного проектирования для создания гибких и расширяемых систем:
  * **S (Single Responsibility):** Один класс должен решать ровно одну задачу (иметь одну причину для изменения).
  * **O (Open/Closed):** Классы открыты для расширения (через полиморфизм), но закрыты для изменения своего исходного кода.
  * **L (Liskov Substitution):** Наследники должны корректно замещать базовые классы без нарушения логики работы программы.
  * **I (Interface Segregation):** Клиенты не должны зависеть от методов интерфейсов, которые они не используют (лучше много мелких интерфейсов).
  * **D (Dependency Inversion):** Модули верхних уровней не должны зависеть от модулей нижних уровней. Оба должны зависеть от абстракций (интерфейсов).

---

### **Глубокие вопросы**

#### **Как принцип подстановки Барбары Лисков (LSP) соотносится с обработкой nullability в Kotlin?**
* **Ответ:** Система типов Kotlin со встроенной поддержкой `nullability` строго защищает принцип подстановки Лисков на этапе компиляции.
  По принципу LSP, переопределенный метод в подклассе не может предъявлять более жесткие требования к входным аргументам (предусловия), чем базовый класс.
  Если базовый класс объявляет метод, принимающий nullable тип `String?`, подкласс при переопределении метода обязан принимать именно `String?`. Если бы подкласс попытался сузить тип до non-nullable `String`, это привело бы к NPE в коде, который работает с базовым классом и передает туда `null`. Котлиновский компилятор считает это нарушением сигнатуры и выдает ошибку компиляции.
* **Пример:**
```kotlin
open class BaseSaver {
    open fun save(data: String?) {} // Базовый контракт готов работать с null
}

class DiskSaver : BaseSaver() {
    // override fun save(data: String) {} 
    // ОШИБКА КОМПИЛЯЦИИ! Нарушение LSP. Наследник не может сузить тип входного параметра до non-null.
}
```

---

### **Каверзные вопросы**

#### **Нарушает ли паттерн «Синглтон» принципы SOLID? Если да, то какие?**
* **Ответ:** Да, практически всегда нарушает:
  1. **Single Responsibility (SRP):** Синглтон решает свою непосредственную бизнес-задачу и одновременно сам управляет собственным жизненным циклом и единственностью своего экземпляра в памяти.
  2. **Dependency Inversion (DIP):** Вызывающий код начинает жестко зависеть от конкретного статического класса синглтона (например, `PreferencesHelper.getInstance().save()`), вместо зависимости от абстрактного интерфейса. Это делает невозможным подмену синглтона на тестовый мок в юнит-тестах.

---

## 12. Дополнительные принципы и GRASP

### **Базовые вопросы**

#### **Что такое DRY, KISS, YAGNI?**
* **Ответ:**
  * **DRY (Don't Repeat Yourself):** Избегайте дублирования логики и знаний в системе. Любая логика должна иметь единственное представление в кодовой базе.
  * **KISS (Keep It Simple, Stupid):** Стремитесь к простоте кода. Сложные многоуровневые решения тяжелее поддерживать и тестировать.
  * **YAGNI (You Aren't Gonna Need It):** Не пишите код и абстракции под требования, которых нет сейчас, но которые "возможно понадобятся в будущем".

---

#### **Что такое GRASP? Назовите ключевые паттерны.**
* **Ответ:** **GRASP** (General Responsibility Assignment Software Patterns) — набор принципов объектно-ориентированного проектирования для назначения обязанностей классам.
  * **Information Expert:** Обязанность должна быть назначена классу, у которого есть максимум информации для ее выполнения.
  * **Creator:** Определяет, какой класс должен создавать объекты другого класса.
  * **Controller:** Координатор системных событий за пределами UI.
  * **Low Coupling / High Cohesion:** Поддержание низкой связанности между классами и высокой внутренней сфокусированности каждого класса на своей задаче.
  * **Pure Fabrication (Чистая выдумка):** Создание искусственного класса (не из предметной области) для сохранения Low Coupling и High Cohesion (пример — `Repository`).
  * **Protected Variations (Защита от изменений):** Изоляция стабильных частей системы от изменений в нестабильных точках через интерфейсы.

---

### **Глубокие вопросы**

#### **В какой ситуации следование принципу DRY (Don't Repeat Yourself) может нанести вред проекту?**
* **Ответ:** На этапе раннего проектирования, когда разработчик совершает **преждевременное обобщение (premature abstraction)**.
  Если две абсолютно разные фичи приложения имеют схожую логику или одинаковый набор полей в UI на старте, объединение их моделей под один общий базовый класс во имя DRY создаст ложную связь. Когда требования к одной фиче изменятся (а это неизбежно), разработчик будет вынужден ставить костыли и разветвления `if/else` в общем базовом классе, ломая вторую фичу. В таких ситуациях дублирование кода является более дешевым и безопасным решением, чем неверная абстракция.

---

### **Каверзные вопросы**

#### **Если мы пишем классы-утилиты (Utils) — какие паттерны GRASP мы нарушаем и используем?**
* **Ответ:**
  * **Используем:** **Pure Fabrication** (Чистая выдумка) — мы создаем искусственные классы, не отражающие реальные объекты бизнеса, чтобы разгрузить другие классы.
  * **Нарушаем:** **High Cohesion** (Высокая связность). Обычно класс-утилита (например, `AppUtils` или `CommonUtils`) превращается в свалку разнородных статических методов, которые никак не связаны между собой логически (и форматирование дат, и шифрование паролей, и проверка подключения к интернету). Правильный подход — заменять такие утилиты мелкими специализированными классами (например, `DateFormatter`, `CryptoProvider`).

# Блок 7: Jetpack Compose

## 13. Основы Compose

### **Базовые вопросы**

#### **Что такое Jetpack Compose?**
* **Ответ:** Это современный декларативный UI-фреймворк от Google для Android. Вместо императивного изменения дерева View-элементов (через `findViewById` или `ViewBinding` и метод `setText`), UI описывается как функция от текущего состояния (`State`). При изменении состояния фреймворк автоматически пересчитывает и обновляет только изменившиеся элементы интерфейса (процесс рекомпозиции).

---

#### **Что такое Recomposition?**
* **Ответ:** Это процесс повторного выполнения `@Composable` функций с целью обновления отображаемого UI при изменении данных (`State`), от которых эти функции зависят. Compose оптимизирует этот процесс, обновляя только те конкретные виджеты, параметры которых реально изменились, и пропуская выполнение тех компонентов, параметры которых остались прежними (интеллектуальный пропуск).

---

### **Глубокие вопросы**

#### **Каким образом компилятор Compose модифицирует Composable-функции на этапе сборки?**
* **Ответ:** Плагин компилятора Compose переписывает сигнатуру всех функций, помеченных аннотацией `@Composable`.
  1. В параметры функции добавляется скрытый аргумент `$composer: Composer` (ссылка на Slot Table — древовидную базу данных UI).
  2. Добавляется целочисленный параметр `$changed` (битовая маска для отслеживания изменений параметров).
  3. В начале тела функции вставляются вызовы регистрации группы в Slot Table (`composer.startRestartGroup()`), а в конце — закрытие группы (`composer.endRestartGroup()`). Это позволяет рантайму Compose позиционировать элементы на экране и пропускать их выполнение при рекомпозиции.
* **Пример:** Что генерирует компилятор под капотом (псевдокод):
```kotlin
// Оригинал
@Composable
fun MessageView(text: String) { Text(text) }

// После компиляции
fun MessageView(text: String, $composer: Composer?, $changed: Int) {
    val composer = $composer.startRestartGroup(12345) // ID группы
    if ($changed and 0b0001 == 0) {
        // Проверяем стабильность: если text не изменился, пропускаем выполнение
        composer.skipToGroupEnd() 
    } else {
        Text(text, composer, 0)
    }
    composer.endRestartGroup()?.updateScope { nextComposer -> MessageView(text, nextComposer, $changed) }
}
```

---

#### **Что такое Slot Table и как Compose использует ее для инкрементальных обновлений?**
* **Ответ:** **Slot Table** — это внутренняя структура данных Jetpack Compose (реализованная как Gap Buffer), которая хранит информацию обо всей структуре пользовательского интерфейса: вызовах Composable-функций, сохраненных значениях `remember`, параметрах и разметке.
  Во время рекомпозиции Compose последовательно идет по Slot Table и сравнивает новые аргументы функций с теми, что были записаны в таблицу на предыдущем шаге. Если аргументы идентичны, Compose перемещает указатель в конец текущей группы, полностью пропуская выполнение кода функции. Если аргументы изменились, Compose обновляет ячейку в Slot Table и запускает рекомпозицию группы.

---

### **Каверзные вопросы**

#### **Что произойдет, если в Composable-функции использовать обычную локальную переменную вместо `mutableStateOf` и пытаться менять ее по клику?**
* **Ответ:** Значение переменной будет меняться в памяти JVM, но на экране ничего не изменится. UI не обновится, так как обычная переменная не имеет механизмов оповещения рантайма Compose. Чтобы запустить процесс рекомпозиции, Compose должен перехватить факт изменения данных. Для этого используются типы `MutableState`, которые при изменении своего поля `value` отправляют сигнал планировщику Compose о необходимости перерисовать все функции, которые читали это свойство.
* **Пример:**
```kotlin
@Composable
fun ClickCounter() {
    var count = 0 // Обычная переменная
    
    Button(onClick = { count++ }) {
        Text("Кликов: $count") // Элемент никогда не обновится на экране!
    }
}
```

---

#### **Почему нельзя использовать `Modifier` со сложными динамическими вычислениями (например, анимацией смещения) без обертки в `graphicsLayer`? Как это влияет на пропуск фаз Compose?**
* **Ответ:** Если мы используем динамическое состояние в обычном модификаторе (например, `Modifier.offset(y = animateDpAsState)`), то при изменении каждого пикселя анимации Compose будет вынужден полностью перезапускать первую фазу — **Composition** (рекомпозицию) и вторую фазу — **Layout** (перерасчет размеров и позиций всех дочерних элементов в дереве). Это создает огромную нагрузку на CPU.
  Использование `Modifier.graphicsLayer { translationY = animatedValue }` переносит анимацию на уровень графического процессора (GPU). Фазы Composition и Layout полностью пропускаются, Compose сразу переходит к фазе **Drawing** (отрисовке измененного слоя на холсте), что гарантирует стабильные 60/120 FPS.
* **Пример:**
```kotlin
@Composable
fun AnimatedItem(scrollState: ScrollState) {
    // ПЛОХО: Вызывает проход Layout для всего дерева на каждый пиксель скролла
    Box(Modifier.offset(y = scrollState.value.dp))

    // ОТЛИЧНО: Координаты макета стабильны, смещение выполняется на уровне GPU
    Box(Modifier.graphicsLayer { translationY = scrollState.value.toFloat() })
}
```

# Блок 8: Android Pitfalls / Errors (Ошибки и подводные камни)

## 14. Утечки памяти (Memory Leaks)

### **Базовые вопросы**

#### **Почему ViewModel не должна хранить ссылку на Activity или Fragment?**
* **Ответ:** Время жизни ViewModel превосходит время жизни Activity и Fragment. При смене конфигурации (повороте экрана) текущий экземпляр Activity полностью уничтожается и создается новый. Если ViewModel сохранит сильную ссылку на старый экземпляр Activity, то сборщик мусора (GC) не сможет очистить этот экземпляр и все связанные с ним View-компоненты из памяти. Это приведет к утечке памяти размером в десятки мегабайт на каждый поворот экрана.

---

### **Глубокие вопросы**

#### **Каков лимит размера системного буфера транзакций Binder и почему попытка сохранить большой объем данных в `SavedStateHandle` или `Bundle` гарантированно роняет приложение?**
* **Ответ:** Системный буфер транзакций IPC Binder имеет жесткий лимит в **1 МБ на весь процесс приложения**. 
  Этот буфер является общим для всех транзакций между вашим приложением и системным процессом OS Android (передача аргументов в `Intent`, запуск новых Activity, сохранение состояния экранов при уходе в фон через `onSaveInstanceState` и `SavedStateHandle`).
  Если попытаться сохранить в `SavedStateHandle` тяжелый список объектов, массив байт или картинку (весом более 400-500 КБ), то при уходе приложения в фон лимит буфера Binder будет мгновенно превышен. Приложение завершится аварийно с исключением `TransactionTooLargeException`.
* **Пример опасного кода:**
```kotlin
class ImageEditorViewModel(private val savedState: SavedStateHandle) : ViewModel() {
    fun saveImage(bitmap: Bitmap) {
        val bytes = bitmap.toByteArray() // Тяжелые данные
        // КРИТИЧЕСКИЙ КРАШ: При сворачивании приложения будет превышен лимит буфера Binder
        savedState["avatar_bytes"] = bytes 
    }
}
```

### **Каверзные вопросы**

#### **Как именно `WeakReference` взаимодействует со сборщиком мусора JVM? Гарантирует ли она немедленное удаление объекта при вызове GC?**
* **Ответ:** Сборщик мусора (GC) автоматически удаляет объект из памяти при первой же сборке, если на этот объект ссылаются **только** слабые ссылки (`WeakReference`). 
  Наличие слабой ссылки не препятствует утилизации объекта. Однако сам вызов GC в JVM является недетерминированным процессом — мы не знаем, в какую именно секунду система решит запустить сборку мусора. До тех пор, пока реальная сборка не произошла, метод `weakRef.get()` будет продолжать возвращать живой объект.
* **Пример использования:** предотвращение утечки Activity в фоновом хелпере.
```kotlin
class BackgroundTask(activity: Activity) {
    // Оборачиваем в слабую ссылку
    private val activityWeak = WeakReference(activity)

    fun onComplete() {
        val act = activityWeak.get() // Возвратит null, если Activity уничтожена и прошел GC
        if (act != null) {
            showUiDialog(act)
        }
    }
}
```

## 15. Ошибки при работе с корутинами и многопоточностью

### **Базовые вопросы**

#### **Что произойдет, если запустить долгую блокирующую задачу на `Dispatchers.Main`?**
* **Ответ:** Запуск блокирующей задачи (чтение файла, тяжелые вычисления, сетевой запрос) на главном потоке блокирует цикл обработки событий (`Looper`). 
  Интерфейс приложения перестает реагировать на действия пользователя (касания, жесты) и перестает перерисовываться (зависает анимация). Если поток заблокирован более 5 секунд, операционная система Android принудительно прерывает работу приложения и показывает пользователю диалоговое окно **ANR (Application Not Responding — Приложение не отвечает)**.

---

### **Глубокие вопросы**

#### **Почему блок `try-catch` вокруг вызова `scope.launch` не перехватывает исключения, возникшие внутри запущенной корутины?**
* **Ответ:** Метод `launch` является **асинхронным неблокирующим строителем**. Он создает корутину, регистрирует ее в планировщике задач и мгновенно возвращает управление (объект `Job`), продолжая выполнять последующий код.
  Внешний блок `try-catch` успешно завершается за доли микросекунд. Сама же корутина начинает физически выполняться на пуле потоков позже. Когда внутри нее возникает исключение, внешнего `try-catch` уже давно нет в стеке вызовов. Исключение уходит вверх к родительскому `Job` и обработчику `CoroutineExceptionHandler`, вызывая падение приложения.
* **Пример ошибки и её исправления:**
```kotlin
// ОШИБКА: Исключение НЕ будет перехвачено, приложение упадет!
try {
    scope.launch {
        throw RuntimeException("Сетевая ошибка")
    }
} catch (e: Exception) {
    println("Поймали!") // Сюда выполнение никогда не зайдет
}

// ПРАВИЛЬНО: Оборачиваем код внутри самой лямбды корутины
scope.launch {
    try {
        throw RuntimeException("Сетевая ошибка")
    } catch (e: Exception) {
        println("Успешно перехвачено: ${e.message}")
    }
}
```

---

### **Каверзные вопросы**

#### **Что произойдет, если вызвать `runBlocking` на главном потоке (Dispatchers.Main), а внутри него запустить корутину на том же `Dispatchers.Main` с вызовом `delay(100)`?**
* **Ответ:** Произойдет взаимная блокировка потока (**Deadlock**).
  1. `runBlocking` блокирует вызывающий поток (в данном случае `Main` поток) до тех пор, пока все задачи внутри его блока не завершатся.
  2. Запущенная внутри корутина вызывает `delay(100)`. Функция приостанавливается, освобождая контекст, и планирует свое возобновление на диспетчере `Dispatchers.Main` через 100 мс.
  3. По истечении 100 мс корутина пытается отправить задачу возобновления в очередь главного потока.
  4. Однако главный поток намертво заблокирован вызовом `runBlocking` и не может обработать очередь задач (`Looper` остановлен).
  Корутина никогда не возобновит работу, а `runBlocking` никогда не завершится. Приложение зависнет навсегда.
* **Пример:**
```kotlin
fun onCreate() {
    // Взаимная блокировка главного потока! Приложение зависает.
    runBlocking {
        launch(Dispatchers.Main) {
            delay(100)
            println("Этот код никогда не выполнится")
        }
    }
}
```
