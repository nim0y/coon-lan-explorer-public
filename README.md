# Архитектура безопасности CoonLan Explorer

Взаимодействие клиента и сервера.

## 0. Обнаружение сервера (Discovery & QR)
Прежде чем начать рукопожатие, клиент должен узнать адрес сервера. Это происходит двумя способами:

1. **mDNS (ZeroConf)**: Сервер вещает о себе в локальной сети. Клиент видит имя устройства и IP.
2. **QR-код**:
   - Сервер генерирует JSON-объект `QrCodeData` (IP, порт, PIN, имя).
   - QR-код является доверенным каналом передачи PIN-кода. Если PIN передан через QR, клиент использует его автоматически для деривации ключей в HKDF.

---

## 1. Этап рукопожатия (Handshake)
Процесс установления защищенного соединения происходит через WebSocket на эндпоинте `/handshake`.

**Поток данных:**
1. **Клиент**:
   - Генерирует временную пару ключей **ECDH** (кривая P-256).
   - Генерирует случайную соль (16 байт).
   - Отправляет `SecureMessage.Handshake(publicKey, salt)` серверу.
2. **Сервер**:
   - Принимает публичный ключ клиента и соль.
   - Проверяет **Rate Limit** (не более 5 попыток за 30 секунд с одного IP).
   - Генерирует свою пару ключей ECDH и свою соль.
   - Вычисляет `Shared Secret` на основе своего приватного ключа и публичного ключа клиента.
   - Использует **HKDF (SHA-256)** для создания сессионного ключа.
     - **Соль для HKDF**: объединение солей клиента и сервера.
     - **Info для HKDF**: строка `"session-key-[PROTOCOL_VERSION]:"` + **PIN-код (PSK)** (если установлен).
   - Генерирует случайный **Bearer Token** (32 символа).
   - Создает сессию, привязанную к токену.
   - Отправляет `SecureMessage.Handshake(serverPublicKey, serverSalt, token)` клиенту.
3. **Клиент**:
   - Вычисляет тот же `Shared Secret` и сессионный ключ (используя введенный пользователем PIN).
   - Сохраняет полученный токен для дальнейших запросов.

---

## 2. Авторизация запросов
Все последующие действия используют полученный токен.

- **HTTP Запросы**: Токен передается в заголовке `Authorization: Bearer <token>`.
- **Загрузка файлов/стриминг**: Токен передается в query-параметре `?token=<token>`.
- **WebSocket (Keep-alive)**: Токен передается в заголовке при установке соединения.

---

## 3. Защита данных в транзите (API)
Передача структурированных данных (списки файлов, статус сервера) защищена дополнительным слоем шифрования.

**Механизм шифрования:**
- **Алгоритм**: AES-256 в режиме **GCM** (Galois/Counter Mode). Это обеспечивает и конфиденциальность, и целостность (защиту от подмены).
- **Структура сообщения**: `SecureMessage.Encrypted(iv, data)`.
  - `iv`: 12 байт (Initialization Vector).
  - `data`: зашифрованная JSON-строка.

**Поток данных API (например, GET /files):**
1. Сервер проверяет токен и права доступа к пути.
2. Сервер сериализует данные в JSON.
3. Сервер шифрует JSON через `CryptoEngine` сессионным ключом.
4. Клиент получает `SecureMessage.Encrypted`, расшифровывает его и парсит JSON.

---

## 4. Передача файлов (Download/Stream)
- Файлы передаются как **raw byte stream** (не шифруются AES-GCM повторно, так как это создало бы огромную нагрузку на CPU).
- Безопасность обеспечивается **проверкой токена** для каждого запроса и жестким **Path Sandboxing** (сервер проверяет, входит ли запрашиваемый путь в список разрешенных папок).

---

## 5. Защитные механизмы и ограничения
- **Session TTL**: Сессия живет 1 час. При неактивности токен аннулируется.
- **WebSocket Presence**: Сервер отслеживает статус WebSocket. Если соединение разорвано, сессия помечается как `DISCONNECTED`. Через 30 секунд после разрыва сессия удаляется.
- **PIN-код**: Является обязательным элементом формирования ключа шифрования (через HKDF). Без знания верного PIN невозможно расшифровать API-ответы, даже имея перехваченный трафик и токен.
- **Network Security**: При смене IP-адреса или идентификатора сети (SSID) сервер выполняет экстренную остановку или перезапуск с обнулением всех сессий.

## Резюме фактов
- **Ключевой обмен**: ECDH (P-256).
- **Деривация ключей**: HKDF (SHA-256).
- **Шифрование данных**: AES-GCM (256 бит).
- **Аутентификация**: Bearer Token + PIN-based PSK.
- **Защита от перебора**: Rate limiting на уровне IP.
